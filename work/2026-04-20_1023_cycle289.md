## **2. Function 1: Deadband Filter**

**Prompt:**
```
Write a Python function `deadband_filter(signal: list[float], bounds: tuple[float, float]) -> list[str]`.
The function takes a list of signal values and a tuple representing the lower and upper safe bounds (P1 channel).
Implement the Deadband Protocol:
- P0 (Negative Space): If value is NaN or infinite, return "ERROR".
- P1 (Safe Channel): If value is within bounds (inclusive), return "SAFE".
- P2 (Optimize): If value is outside bounds, determine if it's above ("HIGH") or below ("LOW") the channel.
Return a list of strings the same length as the input signal.
```

### **DeepSeek-chat Output:**
```python
def deadband_filter(signal: list[float], bounds: tuple[float, float]) -> list[str]:
    """
    Apply Deadband Protocol to a signal.
    P0: Detect invalid values (NaN, infinity).
    P1: Check if within safe bounds.
    P2: Classify out-of-bounds values as HIGH or LOW.
    """
    lower, upper = bounds
    result = []
    for val in signal:
        # P0: Map negative space
        if not isinstance(val, (int, float)) or math.isnan(val) or math.isinf(val):
            result.append("ERROR")
            continue
        # P1: Find safe channel
        if lower <= val <= upper:
            result.append("SAFE")
        # P2: Optimize within channels (classify deviation)
        elif val > upper:
            result.append("HIGH")
        else:  # val < lower
            result.append("LOW")
    return result
```

### **Groq Llama-3.3-70b-Specdec Output:**

```python
def deadband_filter(signal, bounds):
    lower_bound, upper_bound = bounds
    outcomes = []
    for sample in signal:
        # P0: invalid data check
        if not isinstance(sample, (int, float)):
            outcomes.append("ERROR")
            continue
        try:
            if math.isnan(sample) or math.isinf(sample):
                outcomes.append("ERROR")
                continue
        except TypeError:
            outcomes.append("ERROR")
            continue
        # P1: in safe channel?
        if lower_bound <= sample <= upper_bound:
            outcomes.append("SAFE")
        # P2: classify deviation
        elif sample > upper_bound:
            outcomes.append("HIGH")
        else:
            outcomes.append("LOW")
    return outcomes
```

**Comparison:**
*   **Correctness:** Both correctly implement the protocol. DeepSeek's type check `isinstance(val, (int, float))` is more direct. Groq adds a redundant `try/except` block.
*   **Conciseness:** DeepSeek's version is slightly more concise and reads closer to the protocol description (P0/P1/P2 comments). Groq's variable names (`sample`, `outcomes`) are clear but slightly more verbose.
*   **Adherence to Spec:** DeepSeek includes a docstring explaining P0/P1/P2, which is useful for training tiles. Groq does not.

```python
import datetime

def validate_tile(tile):
    if not isinstance(tile, dict):
        return (False, ["Input must be a dictionary"])
    
    errors = []
    # required fields
    required_fields = ["id", "agent_role", "task", "content", "timestamp"]
    for rf in required_fields:
        if rf not in tile:
            errors.append(f"Missing required field: {rf}")
    
    # type checks
    if "id" in tile and not isinstance(tile["id"], str):
        errors.append("Field 'id' must be a string")
    if "agent_role" in tile and not isinstance(tile["agent_role"], str):
        errors.append("Field 'agent_role' must be a string")
    if "task" in tile and not isinstance(tile["task"], str):
        errors.append("Field 'task' must be a string")
    if "content" in tile and not isinstance(tile["content"], str):
        errors.append("Field 'content' must be a string")
    if "timestamp" in tile and not isinstance(tile["timestamp"], datetime.datetime):
        errors.append("Field 'timestamp' must be a datetime object")
    
    # optional tags
    if "tags" in tile and not isinstance(tile["tags"], list):
        errors.append("Field 'tags' must be a list if present")
    
    valid = len(errors) == 0
    return (valid, errors)
```

**Comparison:**
*   **Correctness:** Both are correct. DeepSeek uses a dictionary to map fields to types, making the code more maintainable if fields change. Groq uses separate `if` statements for each type check, which is more explicit but repetitive.
*   **Conciseness:** DeepSeek's loop over `required` is more concise. Groq's version includes an initial check that `tile` is a dict, which is a good defensive practice.
*   **Adherence to Spec:** DeepSeek's code structure reflects a more systematic validation pattern. Groq's is straightforward and easy to debug. DeepSeek's commented-out code about extra fields shows consideration for spec evolution.
