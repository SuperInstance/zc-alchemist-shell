## **2. Function 1: Deadband Protocol Validator**

### **DeepSeek-chat Output:**
```python
def validate_deadband_sequence(actions):
    """
    Validate a sequence of actions against the Deadband Protocol priority rules.
    
    The Deadband Protocol requires:
    P0: Map negative space (what NOT to do)
    P1: Find safe channels (where you CAN be)
    P2: Optimize within channels (best path)
    Strict priority: Never skip to P2 before P0 and P1 are addressed.
    
    Args:
        actions (list of str): List of action tags, e.g., ['P0', 'P1', 'P2', 'P1']
    
    Returns:
        tuple: (bool, str) - True if valid, False otherwise; message explaining violation.
    """
    valid_tags = {'P0', 'P1', 'P2'}
    seen = set()
    
    for action in actions:
        if action not in valid_tags:
            return False, f"Invalid action tag: {action}"
        
        # Check strict ordering: P0 before P1 before P2
        if action == 'P1' and 'P0' not in seen:
            return False, "P1 encountered before P0"
        if action == 'P2' and ('P0' not in seen or 'P1' not in seen):
            return False, "P2 encountered before P0 and P1"
        
        seen.add(action)
    
    return True, "Sequence valid"

# Example usage
if __name__ == "__main__":
    test_sequence = ['P0', 'P1', 'P2', 'P1']
    is_valid, msg = validate_deadband_sequence(test_sequence)
    print(f"Valid: {is_valid}, Message: {msg}")
    # Output: Valid: True, Message: Sequence valid
    
    invalid_sequence = ['P1', 'P0', 'P2']
    is_valid, msg = validate_deadband_sequence(invalid_sequence)
    print(f"Valid: {is_valid}, Message: {msg}")
    # Output: Valid: False, Message: P1 encountered before P0
```

```python
def deadband_protocol_check(actions):
    """
    Check if a list of action phases follows the Deadband Protocol.
    
    Protocol rules:
    - P0 must occur before P1
    - P1 must occur before P2
    - All phases are optional, but if present must follow order
    - Only valid phases are 'P0', 'P1', 'P2'
    
    Args:
        actions (list): List of phase identifiers (strings)
    
    Returns:
        dict: {'valid': bool, 'error': str or None}
    """
    allowed = {'P0', 'P1', 'P2'}
    phase_order = {'P0': 0, 'P1': 1, 'P2': 2}
    last_phase_index = -1
    
    for i, phase in enumerate(actions):
        if phase not in allowed:
            return {'valid': False, 'error': f'Invalid phase {phase} at position {i}'}
        
        current_index = phase_order[phase]
        if current_index < last_phase_index:
            return {'valid': False, 'error': f'Phase {phase} out of order at position {i}'}
        
        last_phase_index = current_index
    
    return {'valid': True, 'error': None}

# Example
if __name__ == '__main__':
    # Test case 1: valid order
    result = deadband_protocol_check(['P0', 'P1', 'P2'])
    print(result)  # {'valid': True, 'error': None}
    
    # Test case 2: invalid order
    result = deadband_protocol_check(['P1', 'P0'])
    print(result)  # {'valid': False, 'error': 'Phase P0 out of order at position 1'}
    
    # Test case 3: with repeats
    result = deadband_protocol_check(['P0', 'P1', 'P1', 'P2'])
    print(result)  # {'valid': True, 'error': None}
```

### **Comparison:**
- **Correctness:** Both handle basic ordering. DeepSeek's version explicitly checks for missing prior phases (P2 before P0/P1), while Groq's uses index comparison which also catches out-of-order but may allow P2 without P1 if indices are monotonic (e.g., `['P0', 'P2']` fails in DeepSeek, passes in Groq). DeepSeek's check is stricter and matches doctrine better.
- **Clarity:** DeepSeek includes full protocol description in docstring. Groq's docstring is concise but less explanatory.
- **Completeness:** DeepSeek validates tags; Groq does too. Edge cases: Groq allows repeats (P1, P1), DeepSeek allows any repeats after prerequisites are met. Both acceptable.
- **Fleet Relevance:** DeepSeek's logic mirrors "Never skip to P2" more directly.

def score_training_tile(tile_text, max_score=10):
    """
    Score a training tile based on fleet criteria.
    
    Criteria (equal weight):
    1. Clarity: Is the writing clear and unambiguous?
    2. Specificity: Does it contain concrete details, not vague claims?
    3. Usefulness: Would this help train an ensign model?
    
    Args:
        tile_text (str): The text content of the tile.
        max_score (int): Maximum score per criterion (default 10).
    
    Returns:
        dict: Scores per criterion and total (0-30).
    """
    # Simple heuristic scoring - in practice would use more sophisticated NLP
    words = tile_text.split()
    sentences = tile_text.count('.') + tile_text.count('!') + tile_text.count('?')
    
    # Clarity: longer sentences reduce clarity
    clarity = max_score
    if sentences > 0:
        avg_words_per_sentence = len(words) / sentences
        if avg_words_per_sentence > 25:
            clarity = max_score * 0.5
        elif avg_words_per_sentence > 15:
            clarity = max_score * 0.8
    
    # Specificity: count of numbers, proper nouns, specific tech mentions
    specific_indicators = sum([
        tile_text.count('P0'), tile_text.count('P1'), tile_text.count('P2'),
        tile_text.count('plato'), tile_text.count('ensign'),
        sum(c.isdigit() for c in tile_text) // 10  # rough count of numbers
    ])
    specificity = min(max_score, specific_indicators * 2)
    
    # Usefulness: presence of actionable terms
    useful_terms = ['function', 'test', 'example', 'code', 'protocol', 'validate']
    usefulness = sum(term in tile_text.lower() for term in useful_terms) * (max_score / len(useful_terms))
    
    return {
        'clarity': round(clarity, 1),
        'specificity': round(specificity, 1),
        'usefulness': round(usefulness, 1),
        'total': round(clarity + specificity + usefulness, 1)
    }
