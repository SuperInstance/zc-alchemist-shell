def validate_deadband_protocol(steps: list[str]) -> tuple[bool, str]:
    """
    Validates a sequence of steps against the Deadband Protocol (P0->P1->P2).

    The Deadband Protocol requires phases in strict order:
    P0: Map negative space
    P1: Find safe channels
    P2: Optimize within channels
    All three must be present, in order, with no skipping.

    Args:
        steps: List of step strings (e.g., ['P0', 'P1', 'P2']).

    Returns:
        Tuple of (is_valid: bool, message: str).
    """
    # Define the required phase sequence
    REQUIRED = ["P0", "P1", "P2"]

    # Handle empty or too-short input
    if not steps:
        return False, "Empty step list. Protocol requires P0, P1, P2 in order."
    if len(steps) < len(REQUIRED):
        return False, f"Only {len(steps)} step(s) provided. Protocol requires 3 phases."

    # Check for required phases in order
    req_idx = 0
    for step in steps:
        # Normalize step: strip whitespace, uppercase
        normalized = step.strip().upper()
        if normalized == REQUIRED[req_idx]:
            req_idx += 1
            if req_idx == len(REQUIRED):
                # Found all required phases in order
                break

    if req_idx == len(REQUIRED):
        return True, "Protocol valid: P0->P1->P2 found in correct order."
    else:
        missing = REQUIRED[req_idx]
        return False, f"Protocol violation. Expected '{missing}' phase not found in sequence."
```

    """
    Validate if the given steps follow the Deadband Protocol.

    The Deadband Protocol requires:
    - P0 (map negative space) must come first.
    - P1 (find safe channels) must come after P0.
    - P2 (optimize within channels) must come after P1.
    - All three phases must be present in order.

    Args:
        steps (list[str]): List of step strings.

    Returns:
        tuple[bool, str]: (is_valid, message)
    """
    expected_phases = ["P0", "P1", "P2"]
    phases_found = []

    for step in steps:
        step_clean = step.strip().upper()
        if step_clean in expected_phases:
            phases_found.append(step_clean)

    # Check if we have all phases
    if len(phases_found) < 3:
        return False, f"Missing phases. Found: {phases_found}"

    # Check order
    for i in range(len(phases_found) - 1):
        current_phase = phases_found[i]
        next_phase = phases_found[i + 1]
        if expected_phases.index(current_phase) > expected_phases.index(next_phase):
            return False, f"Phase order violated. {current_phase} should not come after {next_phase}"

    # Check for duplicates or extra phases? The spec doesn't forbid extra steps between.
    # But we must ensure at least P0, P1, P2 appear in order.
    # Let's verify the sequence contains the expected phases in order as a subsequence.
    seq_idx = 0
    for phase in phases_found:
        if phase == expected_phases[seq_idx]:
            seq_idx += 1
            if seq_idx == len(expected_phases):
                break

    if seq_idx == len(expected_phases):
        return True, "Deadband Protocol validated successfully."
    else:
        return False, f"Phases not found in correct order. Expected sequence {expected_phases}"
```

### **Analysis for Function 1:**
*   **Correctness:** Both are logically correct, but they interpret the requirement differently.
    *   **DeepSeek:** Implements *strict sequence matching*. It scans the original `steps` list for the exact sequence `["P0", "P1", "P2"]`, allowing other steps to be interspersed. This matches the "no skipping" rule literally.
    *   **Groq:** First filters `steps` to only protocol phases (`phases_found`), then checks if the filtered list contains the sequence. This is more permissive but could miss "skipping" if non-phase steps separate the required ones. Its logic is more complex and slightly ambiguous.
*   **Robustness:** Both handle empty/short lists. DeepSeek's edge case messages are more specific.
*   **Clarity:** DeepSeek's algorithm is simpler and its docstring more closely mirrors the fleet's description. Groq's docstring and comments are good, but its implementation has redundant checks.
*   **Pythonicity:** Both are fine. DeepSeek uses an early break, which is efficient.

**Verdict:** DeepSeek's implementation is more aligned with a strict reading of the protocol. Groq's is functional but over-engineered.
