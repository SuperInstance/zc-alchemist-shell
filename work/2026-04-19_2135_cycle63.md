## **2. Function 1: `parse_log_line`**

### **DeepSeek-chat version**
```python
import re
from datetime import datetime

def parse_log_line(log_line: str) -> dict:
    """
    Parse a structured log line in format:
    'YYYY-MM-DD HH:MM:SS [LEVEL] message'

    Returns dict with keys: timestamp, level, message.
    Returns empty dict if line does not match.
    """
    pattern = r'(\d{4}-\d{2}-\d{2} \d{2}:\d{2}:\d{2}) \[(\w+)\] (.+)'
    match = re.match(pattern, log_line)
    if not match:
        return {}

    timestamp_str, level, message = match.groups()
    try:
        timestamp = datetime.strptime(timestamp_str, "%Y-%m-%d %H:%M:%S")
    except ValueError:
        timestamp = timestamp_str  # fallback to string

    return {
        "timestamp": timestamp,
        "level": level.upper(),
        "message": message.strip()
    }
```

### **Groq Llama-70b version**
```python
import re
from datetime import datetime

def parse_log_line(log_line):
    """
    Parses a log line with format: 'YYYY-MM-DD HH:MM:SS [LEVEL] message'
    Returns a dictionary with 'timestamp', 'level', 'message'.
    If format invalid, returns None.
    """
    regex = r'^(\d{4}-\d{2}-\d{2} \d{2}:\d{2}:\d{2}) \[([A-Za-z]+)\] (.*)$'
    m = re.match(regex, log_line)
    if m is None:
        return None

    ts_str, lvl, msg = m.groups()
    try:
        ts = datetime.strptime(ts_str, '%Y-%m-%d %H:%M:%S')
    except Exception:
        ts = ts_str

    return {
        'timestamp': ts,
        'level': lvl.upper(),
        'message': msg.strip()
    }
```

### **Comparison**
- **Correctness:** Both correctly match the pattern and handle datetime conversion.  
- **Readability:** DeepSeek adds a docstring with return‑value description; Groq uses a simpler docstring. DeepSeek names variables more explicitly (`timestamp_str` vs `ts_str`).  
- **Robustness:** DeepSeek returns empty dict on failure; Groq returns `None`. Empty dict may be safer for downstream dict‑access.  
- **Fleet alignment:** Neither uses fleet‑specific patterns; both are generic utilities.  

**Verdict:** DeepSeek’s version is slightly more defensive (empty dict) and has clearer variable names. Groq’s is functionally equivalent but uses `None` for errors.

## **3. Function 2: `deadband_filter`**

### **DeepSeek-chat version**
```python
from typing import List, Tuple

def deadband_filter(readings: List[float], deadband: float = 0.1) -> List[Tuple[float, bool]]:
    """
    Apply Deadband Protocol to a series of readings.
    Returns list of (reading, changed) where changed is True if the reading
    moved outside the deadband relative to last changed reading.
    """
    if not readings:
        return []

    filtered = []
    last_reported = readings[0]

    for r in readings:
        if abs(r - last_reported) >= deadband:
            last_reported = r
            filtered.append((r, True))
        else:
            filtered.append((r, False))

    return filtered
```

### **Groq Llama-70b version**
```python
def deadband_filter(values, deadband=0.1):
    """
    Implements deadband filtering: only report a value if it differs from the
    last reported value by at least deadband.
    Returns list of (value, changed) pairs.
    """
    if len(values) == 0:
        return []

    result = []
    last_reported = values[0]
    for v in values:
        diff = abs(v - last_reported)
        if diff >= deadband:
            last_reported = v
            result.append((v, True))
        else:
            result.append((v, False))
    return result
```

### **Comparison**
- **Correctness:** Both implement the same logic correctly.  
- **Readability:** DeepSeek uses type hints (`List[float]`, `Tuple[float, bool]`); Groq does not. DeepSeek’s docstring explains the fleet’s Deadband Protocol context.  
- **Robustness:** Both handle empty lists.  
- **Fleet alignment:** DeepSeek explicitly references “Deadband Protocol” in the docstring, linking to fleet doctrine. Groq’s version is generic.  

**Verdict:** DeepSeek’s version is more aligned with fleet terminology and includes type hints, which improve maintainability. Groq’s version is functionally correct but less documented.

## **4. Function 3: `validate_github_webhook`**

### **DeepSeek-chat version**
```python
import hmac
import hashlib

def validate_github_webhook(payload_body: bytes, signature_header: str, secret: str) -> bool:
    """
    Validate a GitHub webhook payload using HMAC-SHA256.
    signature_header format: 'sha256=...'
    Returns True if valid, False otherwise.
    """
    if not signature_header.startswith("sha256="):
        return False

    expected_signature = signature_header[7:]  # remove 'sha256='
    mac = hmac.new(secret.encode(), payload_body, hashlib.sha256)
    calculated_signature = mac.hexdigest()

    return hmac.compare_digest(calculated_signature, expected_signature)
```

### **Groq Llama-70b version**
```python
import hmac
import hashlib

def validate_github_webhook(payload, sig_header, secret_key):
    """
    Validates GitHub webhook signature.
    sig_header should look like 'sha256=hexdigest'.
    secret_key is the webhook secret.
    Returns boolean.
    """
    if not sig_header or not sig_header.startswith('sha256='):
        return False

    received_sig = sig_header[7:]
    computed = hmac.new(
        secret_key.encode('utf-8'),
        payload,
        hashlib.sha256
    ).hexdigest()

    return hmac.compare_digest(computed, received_sig)
```

### **Comparison**
- **Correctness:** Both correctly implement HMAC‑SHA256 validation with `compare_digest`.  
- **Readability:** DeepSeek uses more explicit parameter names (`payload_body`, `signature_header`); Groq uses shorter names (`payload`, `sig_header`). Both docstrings are clear.  
- **Robustness:** Both check the `sha256=` prefix. DeepSeek adds a type hint for `bytes` payload.  
- **Fleet alignment:** Neither uses fleet‑specific code; this is a generic security utility.  

**Verdict:** Nearly identical. DeepSeek’s parameter names are slightly more descriptive, and it includes a `bytes` hint for the payload.
